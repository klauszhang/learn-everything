## Every turn re-reads everything

When you send a message to Claude Code, the model doesn't pick up where it left off. It reads the whole thing from scratch: the system prompt, every tool definition, every file that's been loaded, and every message in the conversation so far — then your new message on top.

That's not a bug. It's how transformers work. The model has no persistent memory between requests; its only input is the full text you hand it.

The shaded segments are identical across turns. That's a lot of work being re-done on every request.

## Prompt caching is the workaround

Claude Code uses a feature called **prompt caching**. Claude Code itself (not you) automatically marks "cache breakpoints" at stable points in the request. When the next request arrives with the same prefix up to one of those breakpoints, the model skips re-processing that prefix and reuses the computation from the first time through.

The result: the system prompt, tool definitions, and any files that haven't changed are processed once and reused on every subsequent turn. Only your new message and the latest outputs actually get processed from scratch.

This can cut inference cost dramatically — and speed things up for long conversations. But whether a given turn gets a cache hit depends on whether the prefix matches exactly, token by token.

## What you need to understand transformers to know

That "exact token-by-token match" rule isn't arbitrary. It follows directly from how transformers process text.

### Part I — Transformers & the cache

Seven chapters that build the mental model from tokens up to the prompt cache:

- **Ch 1 — Tokens:** Text gets chopped into tokens. One changed character can shift token boundaries and invalidate the cache.
- **Ch 2 — Embeddings:** Each token maps to a fixed vector. Same token in, same vector out — at the input layer.
- **Ch 3 — Attention:** Tokens "look at" each other. The computation for token N depends on tokens 1…N−1.
- **Ch 4 — Layers:** Many stacked attention + FFN blocks. Each layer builds on the last.
- **Ch 5 — Generation:** The model produces one token at a time, appending each to the input for the next step.
- **Ch 6 — KV cache:** The internal optimization that avoids re-processing the prefix on every generation step.
- **Ch 7 — Prompt cache:** The product feature built on the same idea — but across separate API requests, with its own TTL, pricing, and invalidation rules.

By the end of Part I you'll know exactly what bytes are reused, what makes a cache hit, and what silently blows it cold.
