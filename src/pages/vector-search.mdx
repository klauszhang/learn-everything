## First: two things that share a name

Ch 2 introduced **embedding-layer embeddings**: the model looks up each token ID in a fixed table and retrieves a vector. Token 4721 (`"king"`) always returns the same 12,288-dimensional row — no context, no history, just a table read. That vector then feeds into the first attention layer, where it immediately begins mixing with neighboring tokens.

**Retrieval embeddings are a completely different object.** A separate, purpose-built model — called an *embedding model* — reads your entire query or document all at once and collapses it into *one* vector. That single vector represents the meaning of the whole text, not any individual token. It has a different size (commonly 1,024 dimensions, not 12,288), a different purpose (similarity search, not attention input), and a different lifetime (you can store it in a database and query it months later).

<div class="vs-table-wrap">
| Property | Ch 2: per-token embedding | This chapter: retrieval embedding |
|---|---|---|
| Produced by | The main LLM at inference time | A separate, dedicated embedding model |
| Granularity | One token | One sentence / chunk / document |
| Size (typical) | 4,096–12,288 dims | 256–2,048 dims |
| Lives in | Temporary inference buffers | A vector database or index |
| Consumer | The next attention layer | A nearest-neighbor search |
</div>

When you read about "embeddings" in the context of search and retrieval, this is the kind they mean. Same word, different object.

## Meaning as direction

The embedding model reads your text — say, "What happens if I cancel my subscription?" — and produces a list of numbers, one per dimension. Texts that carry similar meanings end up with similar lists. More precisely, their vectors *point in the same direction* in that high-dimensional space.

That "same direction" is what cosine similarity measures. Given two vectors **A** and **B**:

```
cosine_similarity(A, B) = (A · B) / (|A| × |B|)
```

The denominator removes magnitude — it doesn't matter if one vector has bigger numbers than the other, only the *angle* between them matters. A score of 1.0 means identical direction, 0.0 means perpendicular (unrelated), and -1.0 means opposite.

In practice, most production embedding models return L2-normalized vectors — every vector has magnitude 1, so the denominator is always `1 × 1 = 1`. Cosine similarity collapses to a plain dot product, which is faster to compute and gives identical results. The demo below uses that shortcut.

## The retrieval loop

There are two phases:

**Offline (build the index).** For each document chunk, call the embedding model to get a vector. Store each `{chunk, vector}` pair. This runs once and is re-done when documents change.

**Online (answer a query).** Embed the query, then find the K chunks whose vectors are closest to the query vector. Return those chunks — typically as context for a language model. At scale, finding nearest neighbors uses clever indexes that avoid comparing against every document; covered in a deeper-dive chapter later.

The query step is fast — milliseconds. The LLM reasoning over the retrieved chunks is the bottleneck, not the retrieval.

## Where dense retrieval fails — and why you should know

Three misconceptions are worth heading off before you rely on semantic search in production.

**"The embedding understands the text."** It doesn't, in any meaningful sense. The embedding model has learned which texts appear in similar training contexts. It encodes those statistical patterns as directions in space. It cannot fact-check, catch contradictions, or reason about logic. Two sentences that say the *opposite* of each other may embed nearly identically because they share the same vocabulary and topic structure: "The drug has no side effects" and "The drug has serious side effects" are both sentences about drug side effects — their vectors may be very close.

**"Cosine score ≥ 0.8 means the result is relevant."** Scores are model- and domain-specific. A general-purpose model might place completely unrelated sentences at 0.7, while a domain-tuned model might place genuinely similar sentences at 0.5. There is no universal threshold. What works reliably is *relative ranking* — take the top K results from your model on your data — not absolute cutoffs. Validate on your own corpus before trusting a number.

**"Dense embeddings always beat keyword search."** For natural-language paraphrase — where your query uses different words than the document — dense retrieval wins. But for rare or specialized terms (`TypeError: cannot unpack non-iterable NoneType object`), precise numbers (`$4.2 million` vs. `$42 million`), and verbatim quotes, keyword search (BM25) is more reliable. Embeddings compress meaning, and numerical precision is the first casualty. Most production systems run both in parallel and merge the results. Don't throw away keyword search.

<CacheCallout label="Chapter takeaway">
  Most of what feels like magic in Claude-powered products — file search, knowledge bases, document Q&A — is just this vector lookup running *before* the model ever processes a single token of your question. The model's reasoning is real, but it can only work with what ends up in its context window. Retrieval is how the right text gets there.

  This is also the first step of **RAG (Retrieval-Augmented Generation)**: embed the query, retrieve the nearest chunks, inject them into the prompt, then generate. The next chapter builds the full pipeline. The quality of the retrieval step determines what the model sees — if the right chunk is not retrieved, the answer will be wrong even if the model reasons perfectly over whatever it did receive.
</CacheCallout>
