In Ch 5 you saw that autoregressive decoding generates one token at a time, and that without any optimization, each new token forces the model to re-read the *entire* prefix — every prior token, through every layer. For a 500-token prefix, step 501 does 500× the work of step 1. That cost compounds fast.

The **KV cache** is the fix baked into every modern inference runtime. The key observation is that, within a single generation, the K and V tensors for any token at any layer never change once that token has been processed. Layer 2's K and V for the word "The" are computed once, and they stay the same no matter how many tokens follow — because those values depend only on inputs that are already frozen.

## How it works: store once, read forever

During **prefill** (processing the user's prompt), the model computes Q, K, and V for every prompt token at every layer — exactly the normal forward pass. But instead of discarding K and V after the attention calculation, the runtime saves them to a memory buffer: the KV cache.

During **decode** (generating the response, one token at a time), each new token only needs to compute its own Q, K, and V at each layer. The K and V for all prior tokens are already in the cache. The attention calculation for the new token just reads the cached K and V, computes the dot products, and moves on. No re-computation.

This is an **intra-request** optimization. It lives in runtime memory — GPU VRAM, typically — allocated for the duration of a single generation. When the request ends, the cache is freed. Nothing persists to disk, and nothing is reused across separate API calls.

## Why the cache grows with depth

Every layer has its own K and V tensors. A 4-layer model with a 100-token context holds 4 × 100 = 400 cached K,V pairs. A production-scale model with 96 layers and a 128,000-token context holds far more — memory pressure from the KV cache is a real constraint in serving systems. But the compute saving more than justifies the memory cost: the alternative (recomputing everything at every step) would make long-context generation impractically slow.

## What this is *not*

The KV cache does not reuse work across different API requests. If you send two separate messages to Claude, the second request starts fresh — the KV tensors from the first are gone. It operates entirely within the memory of one running generation, with no persistence, no cross-request identity, and no awareness of previous conversations.

It also does not change *what* the model computes, only *when*. The output of a generation with a KV cache is bit-for-bit identical to one without — the cache just eliminates redundant recalculation.

  The KV cache is the **engine** — the intra-request optimization that makes long-context decoding tractable. It stores K and V for every past token in runtime memory, lives for exactly one generation, and is freed when the request ends.
