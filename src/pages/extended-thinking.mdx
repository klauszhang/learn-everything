## What is the pause?

Sometimes you send Claude a hard question and a "Thinking…" section appears before the actual answer arrives. What is happening in there?

The short answer: nothing hidden. Claude is generating tokens — the same one-at-a-time autoregressive loop described in Chapter 5 — but those tokens are tagged as `type: "thinking"` rather than `type: "text"`. The same model, the same forward passes, the same decoder. What changes is that the model is explicitly given budget to write out intermediate steps before committing to a final answer.

There is no second model, no parallel inference engine, no inner life inaccessible to the API. Every thinking token appears (or is at minimum billed) as normal output. The mechanism is identical to ordinary generation; the consequence — fewer errors on multi-step tasks — is real.

## The same decoder, a different label

In Chapter 5 terms: **prefill** still runs once (your prompt through all layers in a single pass). **Decode** is where thinking lives. With thinking enabled the decode sequence splits into two labeled phases:

## How the API works — 2026 state

The `thinking` field on the Messages API has three modes:

```json
{ "type": "adaptive" }
{ "type": "enabled", "budget_tokens": 10000 }
{ "type": "disabled" }
```

**`adaptive` is the recommended mode** for current models. Claude decides when and how much to think based on request complexity. You can guide the depth with an `effort` level via `output_config: { effort: "<level>" }`:

| Effort | Behavior |
|--------|----------|
| `max` | Always thinks, no depth constraint |
| `xhigh` | Always thinks deeply (Opus 4.7 only) |
| `high` (default) | Always thinks |
| `medium` | Moderate thinking; may skip on simple queries |
| `low` | Minimizes thinking |

**`enabled` (manual mode)** requires a `budget_tokens` value — the maximum tokens the model may spend on its thinking phase. This mode is deprecated on Opus 4.6 and Sonnet 4.6, and **rejected with a 400 error on Opus 4.7**. It still works on Haiku 4.5 and older legacy models (Opus 4.5, Sonnet 4.5).

A key billing nuance: when `display: "summarized"` is set, you see a condensed summary of the thinking — but you are **billed for the full thinking tokens generated**, not the shorter summary. The visible token count and the billed token count do not match. This is documented behavior, not a billing error.

## When thinking helps vs. hurts

**Thinking earns its cost when the task has state that accumulates across steps:**

- Multi-step math and logic — each step depends on a fact established earlier
- Complex planning — laying out a sequence before executing any of it
- Careful debugging — reading and evaluating a system before proposing a fix
- Constraint satisfaction — an answer that must simultaneously meet several requirements

**Thinking adds latency and cost without benefit when:**

- **Factual recall** — "What is the capital of France?" The model does not gain access to facts it does not already have
- **Simple extraction or formatting** — structured-output tasks are typically one-pass
- **Latency-sensitive UIs** — the full thinking phase must complete before answer tokens begin streaming

**Thinking can actively hurt when:**

- You need guaranteed output format (e.g. strict JSON schema) — the thinking phase does not guarantee format adherence; test carefully
- You are running at scale with `effort: "high"` — at Opus 4.7 rates ($25/MTok for all output tokens), a 16,000-token thinking budget costs $0.40 in thinking charges alone, before a single answer token

The adaptive mode with `effort: "medium"` is a reasonable default for mixed workloads: the model will skip thinking for simple lookups and invoke it for harder tasks.

## Three misconceptions

**1. "Thinking is a separate inner model."**
False. Extended thinking is the same decoder producing tagged tokens. There is no second model, no hidden inference engine, no parallel process. The model runs one forward pass per token regardless of whether thinking is on or off. What changes is the label on the tokens, not the mechanism that generates them.

**2. "More thinking budget always means better answers."**
Documented behavior says no. The API docs note diminishing returns above ~32k tokens, and the model may not use the full budget allocated. Setting a 100k budget on a three-step math problem wastes money and time. The model uses roughly the budget it needs for the problem at hand — which is exactly why adaptive mode exists.

**3. "Hidden thinking is private."**
The `thinking` block is encrypted in the sense that you cannot inject fake thinking and have the model trust it (the API verifies the `signature` field). But the API *does* return the thinking content — as a summary or full text depending on the model and `display` setting. It is not hidden from the API caller. Hiding thinking from your end users is a product decision you make, not a default guarantee.

## Cache interaction — the most important section

This is where extended thinking creates a new silent invalidator, directly extending the list from Chapter 7.

**System prompt cache is insulated.** Changing the thinking parameter — including changing `budget_tokens` — does **not** invalidate system prompt cache entries. A cached system prompt survives a switch from `budget_tokens: 4000` to `budget_tokens: 8000`.

**Message-block cache is not.** Changing `budget_tokens` between requests **invalidates message-level cache entries**. The request that changes the budget looks identical to the previous one from the conversation perspective — same messages, same content — but the cache sees a different parameter and treats it as a miss. This is the same class of surprise as the silent invalidators in Chapter 7: no error, no warning, just a miss and full reprocessing cost.

**Switching thinking modes also breaks message cache.** Toggling between `adaptive` and `enabled` or `disabled` invalidates message-level cache breakpoints. Switching `display` between `"summarized"` and `"omitted"` does **not** — the underlying signature is identical regardless of display mode.

**Practical rule:** If you run a multi-turn conversation with thinking enabled and want to preserve message-block cache hits, do not change `budget_tokens` or toggle the thinking type mid-conversation. Pick a mode and stay in it for the session.

For long thinking-enabled sessions (complex multi-step agentic workflows that span more than a few minutes), the 1-hour TTL is recommended over the default 5-minute TTL — a thinking session that runs longer than the TTL window loses its cached system prompt and must pay the full write cost again. As Chapter 7 explains, the 1-hour tier carries a higher write premium, so it only pays off if the session genuinely extends beyond 5 minutes and the prefix is stable.

<CacheCallout label="Chapter takeaway">
Chapter 7's list of silent cache invalidators now includes a new entry: <strong>changing <code>budget_tokens</code> between requests invalidates message-block cache entries</strong>, even when the conversation content is identical. System prompt cache is insulated — it survives thinking parameter changes. Message-block cache is not. Pick a thinking configuration for your session and keep it stable, for the same reason you keep tool definitions stable: any change that shifts the token sequence at a breakpoint turns a cache hit into a full-cost miss, with no warning.
</CacheCallout>
