## The USB-C port for LLM apps

Before MCP, every AI application had to write its own glue code to connect to every external tool or data source — one custom integration per pair. After MCP, a server you write once works in any compatible host: Claude Code, VS Code Copilot, Cursor, ChatGPT, and anything else that speaks the protocol. The analogy on modelcontextprotocol.io says it plainly: "Think of MCP like a USB-C port for AI applications."

Anthropic created MCP and open-sourced it in late 2024. It is not an Anthropic product — it is an open specification. The protocol's current published version is `2025-03-26`, and the ecosystem now spans dozens of clients and hundreds of third-party servers.

**Host** — the application embedding the LLM. Claude Code is a host. VS Code Copilot is a host. The host creates clients, enforces security policies, handles user consent, and decides what the model sees.

**Client** — a connector that lives inside the host, one per server connection. Each client maintains a stateful session with one MCP server, handles protocol negotiation, and routes messages bidirectionally.

**Server** — a process or remote service that advertises capabilities. A server can be a local subprocess (a Python script, a Node package via `stdio`) or a remote HTTPS endpoint. A server has no visibility into the conversation history and does not know what other servers are connected to the same host.

## The wire: JSON-RPC 2.0

At the transport level, MCP is entirely standard. Every message is a JSON-RPC 2.0 object over a transport. There is nothing proprietary about the wire format.

**Transports.** Two are current: **stdio** (the client launches the server as a child subprocess; messages flow over stdin/stdout) and **Streamable HTTP** (the server runs as an independent process exposed via a single HTTP endpoint that accepts POST requests and can respond with SSE streams for async notifications). A third transport — HTTP+SSE from protocol version `2024-11-05` — is **deprecated** as of `2025-03-26` but still supported by Claude Code for backwards compatibility.

**Initialization handshake.** Before any tool is called, client and server negotiate capability. The client sends an `initialize` request with its protocol version and capabilities. The server responds with the version it will use and what it offers. Then the client sends `notifications/initialized` to signal readiness.

**Method vocabulary.** The spec defines a small set of methods:

| Category | Key methods |
|---|---|
| Tools | `tools/list`, `tools/call` |
| Resources | `resources/list`, `resources/read` |
| Prompts | `prompts/list`, `prompts/get` |
| Sampling | `sampling/createMessage` (server → client) |
| Lifecycle | `initialize`, `notifications/initialized` |

## Three primitives — distinguished by who controls invocation

The spec defines three server-side primitives. They are often conflated. The spec separates them by *who decides when to invoke them*.

**Tools — model-controlled.** Tools are functions the LLM can call. The model decides when and whether to invoke them. On session start, the host calls `tools/list` on each server, receives JSON Schema–described function definitions, and injects them into the `tools` array of every API request. The model emits a `tool_use` block; the host routes it via `tools/call` to the right server; the result returns as a `tool_result` block. The model cannot distinguish an MCP-sourced tool from a built-in one — both arrive as entries in the same array.

**Resources — application-controlled.** Resources are content the host can expose to the model: files, database schemas, web pages — anything with a URI. The *host* (not the model) decides which resources to fetch via `resources/read` and how to inject them into context. The model does not call `resources/read` directly.

**Prompts — user-controlled.** Prompts are named templates the user explicitly invokes — think slash commands. A `prompts/get` request with arguments returns a structured message sequence the host can inject into the conversation. Prompts are the least commonly discussed primitive and the most likely to be absent from a given server.

## The sampling reverse channel

Most MCP introductions skip sampling. It is worth understanding.

In every other primitive, the flow is client → server. Sampling reverses this: a server can send a `sampling/createMessage` request to the *client*, asking the host to run model inference on the server's behalf. A documentation-indexing server that needs to summarize files can use the host's LLM rather than maintaining its own API key. The server sends a `sampling/createMessage` with messages and model preferences; the host processes it — potentially showing a confirmation dialog — and returns the model's response.

Three design choices matter here. First, the server needs no API key. Second, the host maintains control: the spec requires hosts to provide UI for users to review sampling requests. Third, model selection is preference-based: the server expresses hints and priority weights, but the client makes the final selection.

The **tool-defs segment** (amber, above) is where every connected MCP server's tool definitions land — merged into the same `tools` array Claude Code sends on every request. When the cache hits, the work of processing all those tool descriptions is served cheaply from cache. When it misses, every token is reprocessed at full cost.

**Adding a server invalidates the segment.** When you add an MCP server, its tool definitions are appended to the tools segment, changing its token sequence. The existing cache entry no longer matches. The next request is a cold miss — for that segment and everything downstream.

**Remote server operators can trigger this without touching your config.** If you are using a remote MCP server and its operator updates tool definitions — rewrites a description, adds a parameter, releases a new tool — the next time Claude Code fetches the tool list (at session start, or via a `notifications/tools/list_changed` notification), the tools segment changes. Your cache goes cold with no warning. This is Ch 7's silent invalidator with a concrete mechanism: your MCP server redeployed.

**More servers = more invalidation surface.** Ten MCP servers might contribute 3,000–5,000 additional tokens to the tools segment. A larger segment means a larger blast radius: one changed tool invalidates the cache savings from all others in that segment, because the whole segment is one cache unit.

## Three misconceptions

**"MCP is an Anthropic feature."** No. MCP is an open protocol that Anthropic created and open-sourced. The specification is maintained under the `modelcontextprotocol` GitHub organization, not Anthropic's. Confirmed adopters include Claude, ChatGPT, VS Code Copilot, and Cursor. A server you write for Claude Code also works in ChatGPT with no modification.

**"More MCP servers = more capable."** Often the opposite. As the tool list grows, the model faces a harder selection problem. Tool selection errors increase. The model may call a plausible-looking tool instead of the correct one, or be confused by overlapping descriptions from multiple servers. Curate; do not accumulate.

**"MCP servers run in the model."** No. MCP servers are processes — local subprocesses or remote HTTP services. They run on your machine or a remote server. The model never executes code. The model emits a `tool_use` block (structured text); the host reads it, routes it to the right MCP server via `tools/call`, gets the result, and returns it to the model as a `tool_result`. The model has no socket, no subprocess, and no direct connection to anything.

<CacheCallout label="Chapter takeaway">
  MCP servers extend Ch 7's silent-invalidator surface beyond your own code. Every server you connect adds its tool definitions to the cached tool-defs segment — the same segment Ch 7 identified as the second cache breakpoint in every Claude Code request. Any server's tool update, whether you caused it or a remote operator did, invalidates that segment cold. More servers means more tokens in the segment and more independent sources of invalidation. Treat MCP server configurations like production dependencies: pin versions where possible, batch updates to minimize cold sessions, and audit tool description changes before upgrading.
