Two "output shape" concerns follow you everywhere in production: **streaming** and **structured outputs**. They are often discussed in the same breath and just as often confused. This chapter separates them.

**Streaming** is about *when* you see the output. The model generates one token at a time regardless of whether streaming is on — that is the autoregressive decode loop from Ch 5. Streaming means each token is sent to you over an open HTTP connection as soon as it is sampled, instead of waiting for the entire response to finish. Nothing about what is generated changes; only the timing changes.

**Structured outputs** are about *what shape* the output can take. A constraint is placed on the response — it must be valid JSON, or must match a particular schema. There are three very different mechanisms for enforcing that constraint, with very different reliability guarantees.

The two concepts compose cleanly once you hold them separately. You can have either without the other, and both together is the normal production configuration.

## Streaming — what is on the wire

### SSE event types

The transport is **Server-Sent Events (SSE)**: a standard browser API for long-lived HTTP connections where the server pushes data. The client keeps the connection open; the server writes lines in the format `event: <name>\ndata: <json>\n\n`. SSE was chosen over WebSockets because it is simpler, HTTP-native, proxy-friendly, and one-directional — which is all you need for a streaming response.

An Anthropic stream follows a fixed sequence. Here is an abbreviated example of a plain text response:

```
event: message_start
data: {"type":"message_start","message":{"id":"msg_01…","content":[],"stop_reason":null,"usage":{"input_tokens":18,"output_tokens":1}}}

event: content_block_start
data: {"type":"content_block_start","index":0,"content_block":{"type":"text","text":""}}

event: content_block_delta
data: {"type":"content_block_delta","index":0,"delta":{"type":"text_delta","text":"Hello"}}

event: content_block_stop
data: {"type":"content_block_stop","index":0}

event: message_delta
data: {"type":"message_delta","delta":{"stop_reason":"end_turn"},"usage":{"output_tokens":3}}

event: message_stop
data: {"type":"message_stop"}
```

The full event sequence for a tool call adds a `content_block_start` of type `tool_use`, followed by `content_block_delta` events carrying `input_json_delta` — partial JSON string fragments. Extended thinking adds a `thinking`-type block before the text block, with `thinking_delta` events and a final `signature_delta` (a cryptographic integrity token, not user-facing text).

| Event | What to do |
|---|---|
| `message_start` | Initialize accumulators. Record `usage.input_tokens`. |
| `content_block_start` | Create a block accumulator keyed on `index`. Note the `type`: `text`, `tool_use`, or `thinking`. |
| `content_block_delta` (`text_delta`) | Append `delta.text` to the text accumulator. Safe to render to UI. |
| `content_block_delta` (`input_json_delta`) | Append `delta.partial_json` to the JSON accumulator. **Do not parse yet.** |
| `content_block_delta` (`thinking_delta`) | Append `delta.thinking` to the thinking accumulator. |
| `content_block_delta` (`signature_delta`) | Store `delta.signature`. Do not render — it is a cryptographic token. |
| `content_block_stop` | If `tool_use`: call `JSON.parse()` on the accumulated JSON string now. |
| `message_delta` | Read `delta.stop_reason`: `"end_turn"` (done), `"tool_use"` (run the tool), `"max_tokens"` (truncated). |
| `message_stop` | Stream is complete. |

`stop_reason` is the most important value in the whole stream — it is the fork in your application's control flow.

A minimal correct parse loop:

```
initialize: text_acc = {}, json_acc = {}
for each SSE event:
  if content_block_start: create accumulator[index] with block type
  if content_block_delta (text_delta): text_acc[index] += delta.text
  if content_block_delta (input_json_delta): json_acc[index] += delta.partial_json
  if content_block_stop (tool_use block): parsed = JSON.parse(json_acc[index])
  if message_delta: act on delta.stop_reason
```

### TTFT vs. tokens/sec — two different bottlenecks

**Time to first token (TTFT)** is the delay between sending the request and seeing the first token. The bottleneck is the *prefill* phase: the model must process the entire prompt before producing anything. A long system prompt means higher TTFT. Streaming does not improve TTFT.

**Throughput (tokens/sec)** is how fast tokens arrive after the first one. This is the *decode* phase, governed by model size and server hardware. Streaming exposes this progressively rather than buffering the full response.

Streaming is not a compute optimization — it is a delivery optimization. The model generates the same tokens at the same speed either way.

## Structured outputs — three mechanisms

"JSON mode" is not a single thing. It maps to at least three distinct mechanisms with different reliability profiles, failure modes, and costs.

### Mechanism 1 — Prompt-constrained

You instruct the model in a system or user message: *"Return your answer as a JSON object with fields `name` and `score`."* The model attempts to comply. If it does not — preamble text, postamble, wrong field names, trailing commas — `JSON.parse()` throws.

**Reliability:** inconsistent. Degrades with schema complexity and proximity to the token limit. **Use when:** prototyping, or human-consumed output.

### Mechanism 2 — Tool use as structured output

You define a tool whose `input_schema` is exactly the JSON structure you want, then force it with `tool_choice: { "type": "tool", "name": "…" }`. The response `tool_use.input` is already a parsed object — no string to parse. The model is trained on trajectories where the `input` matches the schema.

```json
{
  "name": "extract_contact",
  "description": "Extract structured contact information from text.",
  "input_schema": {
    "type": "object",
    "properties": {
      "name":    { "type": "string" },
      "email":   { "type": "string" },
      "company": { "type": "string" }
    },
    "required": ["name", "email"],
    "additionalProperties": false
  }
}
```

**Reliability:** high. The `strict: true` flag adds a constrained-decoding guarantee on top of the training-based reliability. **Cache note:** tool definitions sit in the cached prefix — a schema change is a silent prompt-cache invalidator (see Ch 7).

### Mechanism 3 — Grammar-constrained decoding (`output_config.format`)

The token sampler itself is restricted. Before sampling the next token, the engine filters out any token that would make the partial output invalid against the compiled JSON Schema grammar. The result is mathematically guaranteed to be schema-conformant.

```json
{
  "output_config": {
    "format": {
      "type": "json_schema",
      "json_schema": { "…your schema…": true }
    }
  }
}
```

The schema is compiled and cached for 24 hours. **Unsupported schema features** (they exceed what can be compiled into the required grammar): recursive schemas, numerical range constraints, string length constraints, most array cardinality constraints. This is also what Outlines (`dottxt-ai/outlines`) and llama.cpp's `--grammar` flag implement for open-source models.

### Comparison

| Platform | Feature | Mechanism | Guarantee |
|---|---|---|---|
| Anthropic (historical) | Prompt instruction | Prompt-constrained | None |
| Anthropic (historical) | Tool-use pattern | Training + schema | High, not absolute |
| Anthropic (2026) | `output_config.format` | Grammar-constrained decoding | Absolute (within schema limits) |
| OpenAI | `json_object` mode | Prompt-constrained | None |
| OpenAI | `json_schema` + `strict: true` | Grammar-constrained decoding | Absolute (within schema limits) |

## Three misconceptions

**"JSON mode = sampler-constrained on Anthropic."** Not historically. The classic Anthropic recommendation was the tool-use pattern — training-based, not sampler-level. Anthropic's `output_config.format` (2026) does use constrained decoding. OpenAI's `json_object` mode does not. "JSON mode" has meant all three mechanisms on different platforms at different times. Ask which mechanism, not just the name.

**"Streaming makes the model faster."** No. Streaming reduces *perceived* time to first result by delivering tokens as they are produced. Total generation time is identical with or without streaming. The model generates the same tokens at the same speed either way.

**"Schema validation = correct content."** Schema validation guarantees shape, not truth. With grammar-constrained decoding, the output matches the schema. It will not necessarily contain correct values. A field typed as `number` can contain any number. A field typed as `string` can contain any string. The model can produce schema-conformant output that is factually wrong or semantically nonsensical. Schema compliance tells you the structure is right; semantic validation is a separate, downstream concern.

## Cache interaction

Two independent caching systems interact when you use structured outputs:

**Prompt cache (Ch 7)** — keyed on the token-level prefix, TTL 5 min or 1 hour. Tool definitions sit in the cached prefix. Using the tool-use pattern for structured output means schema changes blow the prompt cache cold from the Tool defs breakpoint forward — silently, with no API warning.

**Grammar cache** — keyed on schema content, TTL 24 hours. Used only by `output_config.format`. A schema change invalidates the grammar cache but not the prompt cache. You can have a prompt-cache hit and a grammar-cache miss simultaneously: no extra token charges, but compilation latency on the next request.

The two caches have different TTLs, different invalidation triggers, and different cost implications. A tool-definition schema edit costs you tokens; an `output_config` schema edit costs you latency.

<CacheCallout label="Chapter takeaway">
  On Anthropic, "force JSON" is historically built on tool use: the output schema is a tool definition sitting in the cached prefix, and <code>tool_choice</code> forces a structured <code>tool_use</code> block. The 2026 <code>output_config.format</code> mechanism is a separate grammar-based path. Either way, <strong>schema edits silently invalidate caches</strong> — tool-use schema edits blow the prompt cache (Ch 7); <code>output_config</code> schema edits blow the 24-hour grammar cache. Treat your output schema like your system prompt: stable content you change deliberately, not iteratively.
</CacheCallout>
