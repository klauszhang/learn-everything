A transformer isn't a single attention operation — it's a **stack of identical blocks**, each one refining the representations that come in from below. Ch 3 showed you what one attention layer does. Now we zoom out to see many of them working together.

## The residual stream

Picture a vertical channel of data flowing upward through the model. At the bottom, the input embeddings enter. At the top, the final representations are used to predict the next token. At every layer in between, the same two operations run: **attention** (tokens look at each other) and an **FFN** (feed-forward network, a pair of matrix multiplications with a nonlinearity in between).

Each operation produces an update, and that update is **added** to whatever was already flowing up the stream — this is the *residual connection*. The stream is never replaced; it accumulates small corrections from each layer. That additive design is what lets gradient signals flow cleanly during training and what allows later layers to build on, rather than overwrite, the work of earlier ones.

## Stacked blocks

Most large language models have dozens to nearly a hundred of these blocks — the exact count is one of the main axes of model scale. Each block is structurally identical: attention heads first, then FFN, both adding to the residual stream.

The diagram below shows six layers (a simplified slice; real models have many more). Each box contains the same internal structure. Data enters from the bottom and emerges transformed at the top.

## What the FFN actually is

Each block has two sub-modules: attention (Ch 3) and **FFN**, the *feed-forward network*. Despite the name, it's just two matrix multiplications with a nonlinearity in between, applied independently to every token:

```
FFN(x) = W₂ · σ(W₁·x + b₁) + b₂
```

where `x` is one token's residual-stream vector and `σ` is a nonlinearity (ReLU, GELU, SwiGLU). Two properties matter:

- **No mixing between tokens.** The FFN sees one token at a time. Token-to-token mixing is attention's job; the FFN's job is "now that you've gathered context, think about it."
- **Up-project, then down-project.** `W₁` expands `d_model` → `d_ffn` (typically 4× wider). `σ` adds nonlinearity. `W₂` projects back to `d_model`. The wide middle is where the bulk of model parameters live — usually ~2/3 of each layer. When people say "70B parameters," most of those are FFN weights.

### Worked example (d_model = 2, d_ffn = 4)

```
x = [1, 0]                          ← one token's residual vector

W₁ = [ 1  0   1  -1 ]               (2 × 4: project up)
     [ 0  1  -1   1 ]
b₁ = [ 0, 0, 0, 0 ]

W₁·x + b₁ = [1, 0, 1, -1]
σ = ReLU  → [1, 0, 1,  0]           ← negatives clipped to zero

W₂ = [ 1  0 ]                       (4 × 2: project down)
     [ 1  0 ]
     [ 0  1 ]
     [ 0  1 ]

FFN(x) = W₂ · [1,0,1,0] = [1, 1]
```

That `[1, 1]` is then *added* to the original `x` via the residual connection before flowing into the next layer. Attention says "look around"; FFN says "now process what you saw."

Mechanistic-interpretability work has tied individual FFN hidden units to recognizable patterns — neurons that fire on French verbs, on hex color codes, on Python list comprehensions. The FFN is where most of the model's stored *knowledge* is thought to live, while attention handles the routing.

## What do different layers actually learn?

This is one of the most active and genuinely open research questions in mechanistic interpretability. It's tempting to say something like "early layers handle syntax, middle layers handle meaning, late layers handle task reasoning" — but that tidy story isn't well-supported by the evidence.

What research does show: **different layers specialize in different ways**, and that specialization is real and measurable. Probing studies can identify information in some layers that isn't present in others. But the mapping from "layer number" to "role" is noisy, architecture-dependent, and often doesn't replicate cleanly across model families. Claims like "layer 5 handles grammar" are not a reliable description of how these models actually work.

## Why depth matters

More layers give the model more opportunities to compose and refine its internal representations. In practice, depth (more layers) and width (larger hidden size) are both levers for model capacity — and both scale independently. A wider model has more dimensions in its residual stream; a deeper one has more transformation steps.

This is also why training very deep networks was historically difficult: without residual connections, gradients would vanish before reaching early layers. The "add to the stream" design is what makes depth practical.

<CacheCallout>
  Every layer in the stack maintains its own **K and V matrices** for every token it processes. That means cache size scales with **layer count × token count × hidden size**. A 32-layer model caches 32 times as much K/V data per token as a single-layer model — and a 96-layer model caches three times that again. This is why the KV cache can be a significant fraction of total GPU memory at scale, and why understanding layers is prerequisite to understanding the cost and benefit of caching.
