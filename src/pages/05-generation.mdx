So far we've seen how the model encodes meaning (embeddings), how tokens look at each other (attention), and how those operations stack up across layers. Now comes the question that ties everything together: **how does the model actually produce text?**

The answer is simpler than you might expect — and the simplicity is exactly what makes caching so valuable.

## One token at a time

Language models don't generate a full sentence in one shot. They predict **one token**, then append it to the sequence, then predict the **next** token — and so on until they decide to stop (a special end-of-sequence token) or hit a length limit. This process is called **autoregressive decoding**.

Think of it like a writer who can only look backward: each new word is chosen by reading everything written so far. The model's "everything written so far" is the entire token sequence: the original prompt plus every token it has already produced.

## Prefill vs. decode — two very different phases

Generation actually has two distinct phases, and they behave very differently inside the model:

**Prefill** is the first pass. The model takes the complete prompt — all of its tokens at once — and runs it through every layer in a single forward pass. This is efficient because all prompt tokens are known upfront and can be processed in parallel. Prefill produces the model's first predicted token, and it also computes a set of internal values (the K and V matrices you saw in Ch 3) for every prompt token at every layer.

**Decode** is everything that comes after. Once the first generated token exists, it gets appended to the sequence, and the model runs another forward pass — but now to predict just the second token. Then a third pass for the third token. *Without any optimization*, each decode step would re-process the entire sequence from scratch — prompt tokens plus all generated tokens so far — through every layer. In practice the model uses a **KV cache** to skip the redundant work; that's exactly what Ch 6 is about.

Here is the critical asymmetry: **prefill runs once; decode runs once per generated token.** A 10-token prompt followed by a 50-token response means 1 prefill pass and 50 decode passes — and each decode pass re-reads those same 10 prompt tokens.

## Why this matters for cost

Every decode step asks the model to re-read every prefix token through every layer. If your prompt is 1,000 tokens and the model generates 200 tokens, those 1,000 prompt tokens get re-processed 200 times — once per decode step. For a model with dozens of layers, that is an enormous amount of repeated arithmetic.

This is the core problem that the **KV cache** solves, and Ch 6 is dedicated entirely to it. The short version: the K and V values computed for each token at each layer don't change once that token is processed. So instead of recomputing them from scratch on every decode step, the model can store them and reuse them. The result is that each decode step only needs to compute K and V for the **one new token**, not for the entire prefix.

<CacheCallout>
Without a KV cache, every new token generated requires re-processing the entire prefix — prompt plus all previously generated tokens — through every layer of the model. The repeated arithmetic scales with sequence length: a 1,000-token prefix means 1,000 tokens of work per decode step, for every step. This is why the KV cache (Ch 6) is not optional in practice — it is the mechanism that makes generation fast enough to be useful. The same principle motivates Claude Code's prompt cache: if large portions of the context repeat across turns, avoiding their recomputation is a dramatic win.
</CacheCallout>
