## Claude Code is thinking — what's actually happening?

You type "fix the failing tests in auth.ts" and Claude Code runs for ninety seconds. File reads appear. An edit flashes by. Tests run. More edits. Tests again. Done.

It looks like Claude is doing something fundamentally different from single-turn chat. It is not. What you are watching is a **tool-use loop**: the model emits a tool call, your harness runs the tool and sends the result back, the model sees the expanded conversation and emits another tool call, and this cycle repeats until the model produces a response with no tool call embedded. Every cycle is one *turn*. Every turn is a new API request. Everything else that gets marketed as "agentic" is layered on top of this.

---

## An agent is iterated tool use

In the tool-use chapter, you saw the single-turn cycle: model emits `tool_use`, harness runs the function, harness returns `tool_result`, model responds. An agent is that cycle running until the model decides to stop — or until the harness cuts it off.

The minimal agent has three parts:

**A loop.** The harness runs `while stop_reason == "tool_use": run_tool(); send_result()`. That is the entire engine.

**A system prompt with stopping criteria.** The model needs to know what "done" looks like. Without explicit stopping instructions, the loop can run indefinitely — each tool result suggests another tool call and the model never converges. "Stop when all tests pass" is better than "stop when you think you're done."

**Harness-enforced budgets.** The model decides when to stop, but it can be wrong. The harness enforces hard ceilings: `max_turns`, wall-clock time, maximum spend. Without these, a misconfigured agent is an open billing tab.

## Workflows vs. agents — who controls the loop?

Anthropic draws a line that matters in practice.

**Workflow:** LLMs and tools orchestrated through predefined code paths. You, the developer, design the control flow. The model fills in each designated step. Predictable, testable, cheaper.

**Agent:** The model dynamically directs its own processes and tool use. You give the model a goal and tools; it decides the steps. More flexible, harder to predict, more expensive to debug.

The distinction is about *who controls the loop*. In a workflow, your code determines what happens next. In an agent, the model's output determines what happens next.

Anthropic's practical advice: "Start with simple prompts, optimize them with comprehensive evaluation, and add multi-step agentic systems only when simpler solutions fall short." Most production "AI agents" are workflows in disguise — and that is often exactly the right call. A workflow is easier to audit, cheaper to run, and predictable enough to test. The payoff from a true agent (model-directed control flow) only materializes when the task is open-ended enough that you genuinely cannot enumerate the steps in advance.

A useful heuristic: can you draw a flowchart for the task before the model runs? If yes, that flowchart should probably be your code, not the model's emergent behavior. If you cannot — because the steps depend on what you discover along the way — then you need a true agent.

## Subagents and isolation

A subagent is a separate model context with its own system prompt and tool set, dispatched by a parent agent via the `Agent` tool. When the parent calls `Agent`, the SDK launches a fresh session — a new conversation with no prior message history — and runs it to completion. The subagent's final message returns to the parent as the tool result.

The parent does not "spawn a process." It emits a `tool_use` block with name `"Agent"`, and the SDK handles the fresh context creation, execution, and result delivery. From the model's perspective, calling a subagent is just calling a tool.

What the subagent receives: its own system prompt, the prompt string from the parent's `Agent` call, and its own tool set. What it does not receive: the parent's conversation history, the parent's tool results, or the parent's system prompt. **The only channel from parent to subagent is the text in the dispatch prompt.** If the subagent needs file paths, error messages, or context from the parent's session, the parent must include that information explicitly.

Subagents help in three ways:

- **Context isolation:** long tool-call transcripts stay inside the subagent. The parent receives a summary, not every intermediate file read. Without subagents, every tool call during exploration accumulates in the parent's context window, and every subsequent API request pays to prefill all of it.
- **Specialization:** each subagent can have a tailored system prompt — a `security-reviewer` with vulnerability-checking guidance, without cluttering the general-purpose agent's instructions.
- **Parallel dispatch:** multiple subagents run concurrently. The orchestrator fires three subagents simultaneously and collects results when all three complete.

One hard constraint: **subagents cannot spawn subagents.** The `Agent` tool must not be included in a subagent's tool list. This prevents unbounded recursion.

**Cost note on parallel subagents:** each parallel branch pays full prefill per turn. If parallel subagents share a system prompt and tool definitions with the parent, those segments can be cache-read across all of them — one cache write, N cache reads. But if each subagent has a different system prompt (security reviewer, test runner, doc generator), they cannot share a cached prefix; each pays its own cache write.

## Cache is the economic substrate

This is the load-bearing section. Without it, long agent runs become untenable.

Every turn in an agent loop is a new API request. The conversation history grows by at least two messages per turn: the model's tool call and the tool result. After ten turns, the history segment is twenty messages larger than at turn one. After thirty turns, sixty messages larger. Each message is tokens; each token in the prefix is prefill compute at every subsequent turn.

Without a prompt cache, a thirty-turn agent run pays to prefill an increasingly long document at every single step. The cost structure is superlinear — each step pays for everything before it.

**The prompt cache is what makes multi-step agents economically viable.** The stable portions of the prefix — system prompt, tool definitions, CLAUDE.md — are cache-read instead of cache-written on each turn. You pay a cache-write premium once per session start; every subsequent turn is a cache read. The amortization benefit accumulates across turns, making longer stable-prefix sessions proportionally cheaper per turn.

**The silent cache invalidator in loops:** if any loop iteration modifies the system prompt or changes tool definitions, the cache goes cold from that point forward. Every subsequent turn pays full prefill — not just on the changed segment, but on everything after the invalidated breakpoint. This is the same rule from Chapter 7, but the damage compounds across every subsequent turn in the run.

A specific trap: an agent that dynamically adds new tool definitions as it discovers available capabilities. Turn 1: 5 tools. Turn 5: 7 tools. Turn 12: 9 tools. Every tool-definition change invalidates the cache. The agent is paying full prefill from the tools segment to the end of history on every turn where the tool list changed — which may be most of them. Design tool sets to be stable. If dynamic tool discovery is needed, the subagent mechanism is the correct isolation: let a subagent handle the dynamic portion in its own fresh context, without polluting the parent's tool list.

**One more detail:** very long agent runs can outlive their cache entries. Cache TTLs (5 minutes by default; 1 hour on the extended tier) mean entries written at turn 1 may have expired by turn 40. A run spanning more than the TTL window will experience cache misses on early segments even if nothing changed.

## Failure modes

<ul class="failure-list">
  <li><strong>Loops without termination.</strong> A model that receives no explicit stopping criteria can call tools indefinitely. Each tool result suggests another tool call; the model is never "done" because it never checks. Fix: explicit termination instructions in the system prompt plus harness-level turn and budget limits as a hard backstop.</li>
  <li><strong>Context blow-up.</strong> Tool results accumulate in conversation history. A file read returns 2,000 tokens; ten file reads is 20,000 tokens that every subsequent turn must prefill. Long agentic runs become expensive and eventually incoherent as the context fills. Mitigations: subagents for subtasks (each gets its own fresh context), automatic compaction (the SDK summarizes older history), and concise tool outputs that return signal rather than raw data dumps.</li>
  <li><strong>Confabulated tool results.</strong> If the model emits a tool_use block and the harness fails to return a corresponding tool_result, the conversation is malformed. A model receiving this malformed history may invent a plausible-sounding result and continue. Always return a tool_result for every tool_use, with is_error: true on failure. Never leave a tool call unanswered.</li>
  <li><strong>Tool-selection bias drift.</strong> When many tools have overlapping descriptions, the model picks one based on its training priors and the description text — and it can pick the wrong one systematically. Too many tools, or too many with similar descriptions, degrades selection quality. Fewer, better-described tools consistently outperform large ambiguous inventories.</li>
  <li class="failure-cache"><strong>Cache thrash.</strong> An agent loop that modifies its system prompt or tool definitions on each turn causes a cache miss on every step. The cost structure inverts: what should be a cheap loop becomes a full-prefill-per-step loop. The tell: a session that "feels slower than expected" despite supposedly stable content. Cause: a mid-loop change to an early segment that invalidated everything after it.</li>
</ul>

## Three misconceptions

<ul class="misconceptions">
  <li>
    <div class="myth">Myth: Agents are smarter</div>
    <div class="reality">It is the same model. More agent turns means more tokens, not more intelligence. The model's capabilities per turn are unchanged. What changes is the opportunity to gather information and course-correct — but those are loop-level properties, not model-level properties. Setting max_turns to 50 on a task that needs two turns is wasteful, not helpful.</div>
  </li>
  <li>
    <div class="myth">Myth: More tools = more capable agent</div>
    <div class="reality">Usually the reverse. Too many tools with similar descriptions degrade tool-selection quality. The Anthropic blog post recommends writing tool descriptions like "docstrings for a junior developer" and testing extensively. A small, well-described tool inventory beats a large, ambiguous one. An agent with fifteen tools that all involve "reading" something will develop systematic preferences for the wrong tool based on superficial features of the request.</div>
  </li>
  <li>
    <div class="myth">Myth: Parallel subagents are free</div>
    <div class="reality">Parallel subagents run concurrently, reducing wall-clock time — but each one pays full prefill per turn. Parallelization trades money for speed. Whether that trade is worth it depends on whether the parallel branches share a cached prefix and whether the time savings justifies the spend. Parallel is faster; it is not cheaper.</div>
  </li>
</ul>

<CacheCallout label="Chapter takeaway">
Claude Code looks like magic until you see the loop. Then it looks like cleverly-chained tool calls, judiciously budgeted, against a cached prefix.

The model does not think harder over the course of an agent run. It thinks once per turn, exactly as it does in a single-turn conversation. What changes is that the harness catches the tool-use signal, runs the tool, hands the result back — and the model gets another turn to look at what happened and decide what to do next.

The magic is the engineering: the loop structure, the termination design, the budget controls, the subagent isolation, and the prompt cache that makes the growing prefix affordable. None of these is model-level capability. All of them are harness-level design choices. Get the harness right and the model's baseline capabilities multiply. Get it wrong and you have an expensive loop that runs until it hits a rate limit.
</CacheCallout>
