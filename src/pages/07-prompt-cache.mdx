## From runtime to persistence

In Chapter 6 you saw how the **KV cache** eliminates redundant work inside a single generation: once a token has been processed at a given layer, its key and value vectors are stored and reused for every subsequent decode step — no recomputation needed. That optimization lives entirely in **runtime memory** during one API call. When the call ends, those saved vectors are gone.

The **prompt cache** is a different beast. It is a *product feature* — not an internal kernel optimization — that persists prefix state **across separate API requests**. When the same prefix appears in the next request, the model skips all the transformer layers for those tokens entirely. Same underlying idea ("don't redo work on the prefix"), but different mechanism, different scope, different lifetime, and — crucially — different invalidation rules. This chapter is about the product feature, not the runtime kernel.

---

## Cache breakpoints

Caching doesn't happen automatically for every token. Developers (or Claude Code itself) mark **cache breakpoints**: explicit signals saying "try to write a cache entry up to this point in the request." The API currently supports **up to 4 breakpoints per request**.

On each request, the server looks for the longest matching cached prefix — the furthest breakpoint where the token sequence exactly matches a stored entry. Everything before that breakpoint is served from cache; everything after is processed fresh. The server also walks back a short distance (a "lookback window") from each breakpoint, so a near-match on the prefix can still yield a partial hit at an earlier position rather than nothing at all.

**You do not manage breakpoints when using Claude Code.** Claude Code handles all of this internally — it places breakpoints at the right positions automatically. (Separately, Anthropic's API also does some **automatic caching** of long stable prefixes even when no explicit breakpoint is set; Claude Code's explicit breakpoints sit on top of that.) But understanding where those breakpoints go is exactly what tells you what will and won't be cached.

---

## What counts as a hit

A cache hit is an **exact token-level prefix match** up to a breakpoint. Not approximately the same. Not semantically equivalent. Byte-for-byte identical tokens in exactly the same order.

This matters more than it sounds:

- A one-character typo fix regenerates different tokens and misses.
- Reordering two tool definitions breaks the match even if both are still present.
- Adding a line to the system prompt shifts every downstream token by one and invalidates everything.

---

## TTL and the two tiers

Cached entries expire. The default TTL is **5 minutes**. An extended **1-hour TTL** is also available.

**Both tiers charge a premium to *write* a cache entry** — writes always cost more per token than plain uncached input. The 1-hour tier carries a noticeably higher write premium than the 5-minute tier, in exchange for the longer lifetime. So the 1-hour tier is not just "5-minute but better": you pay extra up front for entries you intend to read back later. The longer lifetime only becomes a net win when that entry will be read many times before it expires — if your turns are spaced far apart and the prefix is stable, the 1-hour tier pays off. If you pay the write premium and then the prefix changes before the next read, you've paid for nothing.

---

## Pricing direction

Prompt cache pricing follows a simple shape — qualitatively:

- **Cache writes** cost *more* per token than plain uncached input — for *both* tiers. The 1-hour tier costs noticeably more to write than the 5-minute tier.
- **Cache reads** cost *much less* per token than uncached input.

The math works because you pay the write premium once and then amortize it over every subsequent cache read. A prefix read many times relative to writes is a clear net win. A prefix read once or twice barely breaks even — and on the 1-hour tier with the higher write premium, it can be a net loss.

Never treat caching as "free." The write always costs something, and avoiding that cost is why **keeping prefixes stable** is the real optimization.

## Silent cache invalidators

This is the part that trips people up. These changes blow the cache cold **without any visible change to the conversation**:

1. **System prompt edits.** Any modification — even a single added space — shifts every token downstream from the very first breakpoint.
2. **Tool definition changes.** Adding, removing, or rewording any tool description changes the token sequence for the tools segment. Everything after tools goes cold.
3. **Segment order changes.** The cache is positional. If two segments are swapped, no existing entry matches.

The danger is subtle: you might refactor a tool description for clarity, run Claude Code again, and wonder why the session "feels slower." The cache was silently invalidated. No error, no warning — just a miss.

---

## Practical takeaways

**1. Put stable content first, mutable content last.**  
System prompts, tool definitions, and large reference files belong at the top of the context stack. Your latest question and fresh tool outputs belong at the bottom. Claude Code already follows this order internally.

**2. Avoid small edits to early segments.**  
A one-line change to the system prompt or a minor tool description tweak invalidates the entire prefix. Batch those changes or save them for the start of a new session.

**3. Tool and system-prompt churn silently nukes the cache.**  
The cache gives no feedback when it's cold. If you're iterating rapidly on tool definitions during development, expect full reprocessing costs on every turn.

**4. The 5-minute TTL means back-to-back turns benefit most.**  
Quick exchanges in an active session are the ideal use case. Long gaps between turns risk TTL expiry. If you're using the 1-hour tier, weigh the higher write cost against your actual read frequency.

**5. Longer TTL does not mean free.**  
Both tiers charge a write premium above uncached input; the 1-hour tier charges noticeably more than the 5-minute one. Don't opt in to 1-hour unless your session spans more than a few minutes and the prefix is genuinely stable.
