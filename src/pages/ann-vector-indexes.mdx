In the last chapter you saw how embeddings let you find similar documents: the closer two vectors are in embedding space, the closer in meaning the texts they represent. The demo there worked on a dozen documents. Now the question is: what happens when you have ten million?

A cosine similarity check against one stored vector takes a few microseconds. Ten million of them — one per document, once per query — takes somewhere between 200 and 500 milliseconds on a modern CPU. At 10,000 queries per hour that's 100 billion distance computations hourly. No live system can absorb that cost and still respond in the time it takes to press Enter.

The naive fix — faster hardware — doesn't close the gap. Linear scan scales as O(N × D) where N is the document count and D is the vector dimension. Double your hardware budget and you halve your latency; double your corpus and you double it back. You can't hardware-spend your way out of a linear scaling problem. You need a structure that lets you skip most of the corpus.

That structure is an **approximate nearest neighbor (ANN) index**. It accepts a carefully bounded risk of missing the single closest match — "approximate" is doing real work in that name — and in exchange returns answers in milliseconds. The bet it makes: the 2nd or 3rd nearest neighbor is usually close enough to the 1st that you don't notice the miss. For semantic retrieval of natural language, that bet almost always pays off.

## The three-way tradeoff

Every ANN system navigates a triangle. The three corners are **recall**, **latency**, and **memory**. You can tune toward any two; the third suffers.

**Recall@K** measures how many of the true K nearest neighbors you actually get back. You establish ground truth by running an exact brute-force search, then compare what the ANN index returned. If the true top-5 for a query are A, B, C, D, E and the index returns A, B, C, D, F — that's Recall@5 = 0.80 (four of five correct). A system with Recall@10 of 0.95 finds, on average, 9.5 of the 10 true nearest neighbors. The missed one is usually so close in semantic space to what was returned that the difference in answer quality is undetectable.

The table below gives directional estimates. These numbers are illustrative syntheses from FAISS benchmarks and Pinecone's published comparisons — exact values depend on dimensionality, tuning parameters, and hardware.

<div className="ann-table-wrap">

| Approach | Recall@10 (typical) | Query latency, 10M vectors | RAM vs. raw vectors |
|---|---|---|---|
| Exact / Flat | 100% | 200–500 ms | ~1× (vectors only) |
| HNSW | ~90–98% | ~1–5 ms | ~2–4× (vectors + graph) |
| IVF+PQ | ~80–90% | ~5–20 ms | ~0.02–0.05× (compressed) |

</div>

The 80–95% recall range is almost always good enough for semantic retrieval. Going from 95% to 99% recall typically requires 3× more search work, while the improvement in retrieved text quality is usually imperceptible. The difference between the 5th-nearest and 6th-nearest document in semantic space is usually a sentence or two of relevance — not the difference between a correct and wrong answer.

## Exact search: the brute-force baseline

Before approximation, understand what is being approximated.

The **Flat index** is the simplest possible structure: for every query, compute the distance between the query vector and every stored vector, then return the K smallest. No graph, no clustering, no training. Just a flat array of vectors. Recall is 100% by definition — whatever the true K nearest neighbors are, you get them.

Latency is linear in corpus size. A 1M-vector Flat index on CPU might return results in about 50 milliseconds; 10M vectors takes around 500 milliseconds. At 100M vectors, brute force is no longer viable for interactive use.

The Flat index earns its place for corpora below roughly 100,000 vectors where query latency requirements are modest, or for batch processing jobs where queries are infrequent. More importantly, it is the reference implementation — when you want to know a system's "true" recall, you compare against Flat results. Every recall number quoted above was measured this way.

## HNSW: navigating a layered graph

**HNSW (Hierarchical Navigable Small Worlds)** is a graph-based algorithm that, as of 2025–2026, is the dominant index type for latency-sensitive deployments. It is the default or preferred algorithm in Weaviate, Qdrant, Milvus, and pgvector.

The navigation intuition is best explained with a zoom-level analogy. Imagine you're looking for a specific restaurant in an unfamiliar city. Walking block-by-block from your hotel would eventually work — but it could take hours. A smarter strategy: zoom out to a highway map, drive to the right neighborhood, switch to a street-level view, then walk the last block. Each transition refines your search from "coarse and fast" to "fine and slow."

HNSW applies exactly this strategy to a high-dimensional point cloud. When a new vector is inserted, the algorithm assigns it to one or more layers using a randomized distribution: most vectors land only in layer 0, a fraction appear in layer 1, an even smaller fraction in layer 2, and so on.

- **Layer 2 (top, sparse):** a small fraction of all vectors, connected to long-range neighbors across the entire space. Think of these as the "highway network" — fast and coarse.
- **Intermediate layers:** progressively more vectors with shorter-range links. The neighborhood level.
- **Layer 0 (dense):** every vector is here. Links connect only to true local neighbors. This is where precise search happens — walking the final block.

At query time the algorithm enters at the top layer, moves greedily toward the query at each layer, then descends to the next layer and refines. By the time it reaches the dense bottom layer, it has been guided to roughly the right region without checking all 60 dots — it only checks the 10–20 in the nearby corner.

This is where approximation enters: a greedy walk can get stuck in a local optimum. If the graph's edges don't happen to route through the globally closest point, you get a close-but-not-closest result.

### The efSearch knob

HNSW has two main tuning parameters.

**`M` (build-time):** the number of bidirectional edges per node. Higher M means better connectivity, higher recall, but more RAM and slower index construction. Typical values: 8–64.

**`efSearch` (query-time):** the size of the candidate priority queue during search — think of it as "how aggressively to explore before settling on an answer." Raising efSearch improves recall and increases latency. The relationship is sublinear in recall gains: going from efSearch=16 to efSearch=64 might add 20 percentage points of recall; going from efSearch=128 to efSearch=256 might add zero. There is a sweet spot beyond which further exploration costs more than it gains. Most production systems tune to the efSearch value that gives 90–95% recall, then stop.

## IVF+PQ: narrow then approximate

When HNSW's memory cost is prohibitive — the full graph requires more RAM than you have — the classic alternative combines two ideas: **partition the space** to skip most of it (Inverted File index, IVF), then **compress each vector** to reduce the cost of searching within the partition (Product Quantization, PQ).

### IVF: divide and conquer

Before the index accepts queries, it runs a training step (k-means clustering) on a sample of the dataset. The result: a set of cluster centroids that divide the embedding space into regions. Every stored vector is then assigned to its nearest centroid and stored in that centroid's "inverted list."

The zip-code analogy is apt: to find the coffee shop nearest your current location, you don't check every shop in the country — you check the postal codes that surround you. At query time, IVF compares the query to the cluster centroids (a cheap operation: typically 256–4,096 comparisons), selects the `nprobe` closest centroids, and then searches exhaustively only within those clusters.

With nlist=4,096 and nprobe=32, you're searching about 0.8% of a 10-million-vector corpus — roughly 80,000 vectors instead of 10 million.

**The approximation:** if the true nearest neighbor happens to live in a centroid's cell that wasn't among the top `nprobe`, it won't be returned. This is the main recall loss in IVF. Increasing `nprobe` reduces the risk at the cost of more search time.

### PQ: compress each vector to bytes

Even after IVF narrows the search to 80,000 vectors, comparing 80,000 full-precision vectors — each 6,144 bytes for a 1,536-dimensional float32 vector — is expensive. Product Quantization compresses each vector from thousands of bytes to a handful.

PQ works by slicing each vector into sub-vectors. Each sub-vector is replaced by a small integer code: "which of 256 representative patterns (the codebook) is this sub-vector closest to?" For a 1,536-dimensional vector split into 8 sub-vectors, the result is 8 bytes per vector instead of 6,144 — a 768× compression.

Distances are then approximated using precomputed lookup tables rather than full vector arithmetic — dramatically faster. The cost: two vectors that were slightly different may encode to the same code. This lossy approximation is the source of IVF+PQ's lower recall compared to HNSW. The recall penalty is a known, bounded engineering trade-off — not random degradation.

Combined, FAISS reports that an IVF+PQ index runs approximately 92× faster than a brute-force Flat index at matching recall levels, while using roughly 96% less memory. For corpora of hundreds of millions of vectors — where HNSW's graph simply does not fit in RAM — IVF+PQ is often the only viable option.

## What a "vector database" actually is

"Vector database" is over-mystified in product marketing and under-explained in technical writing. The reality is simpler.

A vector database is an ANN index wrapped with the operational machinery needed to use it in a real application:

- **Metadata store.** The index maps vectors to positions. Applications need to know which position corresponds to which document — and know attributes like owner, date, and source URL. A metadata store (typically a key-value store or relational table) provides this mapping.
- **Filter logic.** The ability to narrow vector search with metadata predicates: "nearest to this query *and* owned by user X *and* published after 2024." See the section below on why this is harder than it looks.
- **Persistence.** An ANN index loaded in memory doesn't survive a process restart. Production systems need snapshot management and sometimes write-ahead logging.
- **Replication.** For read-heavy workloads, distributing the index across multiple replicas is standard, which adds consistency considerations.
- **Hybrid scoring.** Many production systems combine vector similarity with BM25 keyword relevance. A document might rank higher because it contains exact keywords from the query even if its vector isn't the closest.

"A vector database is not a magic AI database that understands your content. It is a system that answers one very specific query — 'find the K nearest vectors to this point' — very fast, with metadata filtering and persistence added on top." At small workloads (a few hundred thousand vectors, an existing PostgreSQL deployment), pgvector does everything a specialized vector database does with no additional operational dependency. Specialized systems earn their complexity at high query rates, very large corpora, or demanding filtering workloads.

## Filtered search: the quiet gotcha

In production, users almost never ask for "find me the 10 most semantically similar documents, period." They ask for "find me the 10 most similar documents *that I own*, *published this year*, *from the internal wiki*." Combining metadata filters with ANN search is one of the hardest operational problems in retrieval systems, and most naive implementations get it quietly wrong.

There are three strategies:

**Post-filter (search-then-filter).** Run ANN on the full index, retrieve more candidates than requested (say, top-20), then discard any that fail the metadata predicate. Simple to implement — but with a selective filter (only 1% of documents match), you'd need top-1000 candidates to expect 10 survivors. Many systems will silently return fewer results than requested. The correct answer may exist in the index and be the closest semantic match, but if it doesn't appear in the pre-filter ANN top-K, it is invisible.

**Pre-filter (filter-then-search).** Apply the metadata predicate first — using a conventional index on the metadata columns — to produce a subset of matching document IDs. Then run ANN restricted to that subset. Correctness is guaranteed for the filter; if the filtered subset is small enough, exact search over it is trivially fast.

**In-algorithm filtering.** Modify the ANN traversal itself to skip filtered-out nodes. For HNSW this is the most principled approach, but has a subtlety: the graph was built without knowing which filters would be applied. Skipping nodes during traversal can sever graph paths that would have led to the true nearest neighbor. pgvector 0.8.0 addressed this with `hnsw.iterative_scan`: when a filtered search returns too few results, the system expands the search radius iteratively until enough filter-passing results are found.

**Concrete example.** Alice searches for "Q3 budget projections." Her filter: `owner = "alice" AND year = 2025`. The corpus has 2 million documents; Alice owns 200 of them.

With **post-filter**: ANN runs against all 2 million vectors and returns its top 20. Of those 20, roughly one is Alice's (her 200 documents are 0.01% of the corpus). Post-filtering produces 1 result for a K=5 query. Two of Alice's most relevant documents were ranked 22nd and 31st globally — never reached.

With **pre-filter**: the system first retrieves Alice's 200 document IDs, then runs exact cosine similarity against those 200 vectors. Returns the correct top 5. Latency: negligible — 200 exact comparisons is a millisecond of work.

**The lesson:** for narrow filtered queries, pre-filter + exact search is frequently faster *and* always more correct than post-filter. ANN is valuable when the corpus is huge and the filter is loose; it offers no benefit when the filtered subset is already small enough for brute force.

A system's unfiltered recall does not predict its filtered recall. A system benchmarked at Recall@10 = 0.95 on unfiltered queries can have Recall@10 = 0.20 on queries with high-selectivity filters. Teams typically discover this gap only after user complaints about poor results on their personal documents.

## Three misconceptions to retire

**"HNSW is exact."** HNSW is approximate. It uses a greedy graph traversal that can get stuck in local optima — returning a close neighbor that is not the closest. You can push `efSearch` high enough to approach 100% recall, but at that point the index degrades toward brute force. The word "navigable" in the algorithm name means "can be navigated efficiently," not "guaranteed to find the globally nearest point."

**"Bigger index = better recall."** Index quality determines recall; size is a weak proxy. An HNSW index built with a low `M` parameter (sparse graph) will fail to route searches correctly regardless of corpus size. Recall is a function of tuning parameters relative to dataset characteristics — not a monotonic function of corpus size. An HNSW index tuned for 1M vectors and 95% recall may drop to 88% recall at 10M vectors without retuning, because the graph becomes more complex and greedy walks traverse more hops before getting lost.

**"Vector databases are smarter than regular databases."** Vector databases answer one specific query type efficiently: nearest-neighbor by vector distance. They are not more capable than relational databases for structured queries, joins, aggregations, or transactional workloads. A vector database that doesn't support `GROUP BY` or a transaction with rollback is *less* capable than PostgreSQL for most queries — it is specialized for one task, not superior overall. This misconception is perpetuated by marketing that conflates "AI-native" with "better."

<CacheCallout label="Chapter takeaway">
  When a Claude tool says "I searched 50,000 documents in a fraction of a second," it didn't really search all 50,000. It navigated a carefully pre-built graph — starting with long hops across the semantic space, zooming into the right neighborhood, and then exploring only the local area in detail. It returned results that are very likely the closest ones, but not provably so. The entire architecture rests on a bet: "the 2nd-nearest document is close enough to the 1st that missing the 1st occasionally doesn't matter." In retrieval for language models, that bet almost always pays off. The speed isn't magic — it's structure, pre-computed and carefully tuned.

  This is also the chapter where a recurring pattern in this site becomes explicit. Ch 6 showed how the KV cache trades memory for avoiding redundant computation — reusing stored K/V tensors across decode steps instead of reprocessing the full prefix. ANN indexes make the same kind of trade: precompute a graph or partition structure so that query time skips most of the work. Both accept that "less work now" costs something (memory for the KV cache; graph overhead or quantization error for ANN). Recognizing this pattern — build a structure, pay in memory, save on the critical path, accept bounded inaccuracy — is the mental model that carries forward to any fast system you encounter.
</CacheCallout>
