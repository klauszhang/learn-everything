The model never operates on raw text. After tokenization, each token is just a number — an ID from the vocabulary. The first thing the model does is look that ID up in a large table and retrieve a fixed vector of numbers called an **embedding**. That vector is the model's starting representation of the token.

Because the embedding table is fixed after training, the same token ID always returns the same vector. Token 4721 (`"king"`) maps to exactly the same 12,288 numbers every single time, regardless of what text surrounds it. This is the **embedding layer's determinism**: one token in, one fixed vector out.

## Meaning lives in geometry

Embeddings aren't arbitrary. During training, the model adjusts the table so that tokens with similar meanings end up with similar vectors — meaning their coordinates sit close together in the high-dimensional space. You can visualize this in two dimensions: royalty words cluster near each other, animal words cluster elsewhere, and so on. This geometric structure is where linguistic relationships are encoded.

The diagram below compresses those high-dimensional relationships into 2D. Real embedding spaces have thousands of dimensions, but the clustering intuition holds.

## An important boundary: the embedding layer vs. later layers

Here is the nuance that matters most for the rest of this course: **the "same token → same vector" rule applies only at the embedding layer.** The moment the embedding enters the first attention layer, it gets mixed with the representations of other tokens in the sequence. From that point on, the representation of any token is *context-dependent* — "bank" in "river bank" produces a different internal vector after layer 1 than "bank" in "savings bank," even though both started from the same embedding.

This distinction matters because it defines exactly what is and isn't predictable about the model's internal state before inference begins.

<CacheCallout>
  **Same token ID → same embedding-layer vector, always.** The embedding lookup is a pure table read: no context, no history, no randomness. Token 4721 always retrieves the same row. This determinism — at the embedding layer *and* at every layer above it, when the prefix is held fixed — is the foundation that makes prefix caching possible at the API level.

  **After layer 1, the representation is context-dependent.** Attention mixes each token's representation with every other token in the sequence. From layer 1 onward, what the model "thinks about" a token depends on its neighbors. The prompt cache doesn't store embeddings — it stores the **per-layer K and V tensors** computed for a specific sequence of tokens together (you'll see this in Ch 6). What stays cacheable is the work that depends only on the prefix tokens, not "the model's view of a word" in isolation.
