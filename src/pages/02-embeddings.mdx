## From IDs to vectors

The first thing the model does with each token ID is look it up in a learned
matrix — one row per vocabulary entry, each row a vector of d_model numbers.
Click a token to see the lookup in action:

## Semantic geometry

During training, the model pushes tokens with similar meanings to nearby regions
of the vector space. Hover any point to see its vector and nearest neighbors:

<EmbeddingScatter client:visible />

## Positional encoding

Transformers process all tokens in parallel — they have no built-in sense of
order. Positional encoding fixes this by adding a position-dependent signal to
each embedding before attention runs. Try swapping two tokens to see how
position changes the input vector even when the embeddings stay the same:

<PositionalEncoding client:load />

<CacheCallout>

**Embeddings are the transformer's first learned representation.** The lookup is context-free: the same token ID always retrieves the same vector. Context arrives in the next chapter, when attention mixes these vectors together. The quality of this initial embedding space — how well geometry captures meaning — shapes everything the model can learn on top of it.

</CacheCallout>
