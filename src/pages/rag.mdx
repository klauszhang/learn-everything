## The magic trick — and how it works

Imagine a colleague who has read everything published before their knowledge cutoff — but nothing since. They sometimes confabulate a confident answer when they can't quite remember. Now imagine a second colleague who can sprint to the filing cabinet on your behalf, pull the three most relevant pages, and hand them to the first colleague before they answer. That is RAG: **find relevant text first, then generate from it.**

From a user's perspective the effect feels surprising: "the model can suddenly answer questions about my company's internal wiki without retraining." What's actually happening is a search step you don't see. A vector index — built from that same wiki — finds the paragraphs most likely to answer your question. Those paragraphs are injected into the prompt. The model reads them like a person reads a quote before replying. If retrieval is good, the answer is good. If retrieval misses, the model improvises — and that is where RAG fails.

RAG is not magic. It shifts the quality problem from "what did the model memorize?" to "how good is my retrieval?" Both halves matter equally.

## Why RAG exists: four converging problems

### Stale training data

Language models have a knowledge cutoff: a date after which they know nothing. A model trained through late 2024 cannot tell you what your organization shipped in 2025. RAG sidesteps this by pulling from a live or regularly-updated document store — the model's weights don't change; only the documents do.

The honest caveat: the index can go stale too. RAG pushes the freshness problem from model weights to your document ingestion pipeline.

### Hallucination when knowledge is absent

When a model doesn't know something, it rarely says "I don't know." It generates a fluent, confident-sounding answer from statistical pattern-matching. RAG reduces hallucination by giving the model something true to anchor on. If the retrieved chunk directly answers the question, the model follows it. If it doesn't, the model may still hallucinate — but now there is a source document to check against.

### Fine-tuning is expensive and slow

The alternative to RAG is fine-tuning: retraining the model on your private data. Fine-tuning changes the model's weights — GPU cluster, thousands of dollars, a new deployment, a pipeline that re-runs every time your knowledge base changes. Iteration cycles are days or weeks.

RAG changes documents, not weights. A new document can be retrievable in seconds. Fine-tuning cannot match that iteration speed. (Fine-tuning and RAG are complementary: fine-tuning makes the model *reason* in a domain-specific way; RAG puts specific information in context. They solve different problems.)

### Context windows are large, but not the answer

As of 2026, Claude supports a context window of up to 200,000 tokens. But stuffing an entire knowledge base into every request isn't a solution: every input token is billed, time-to-first-token scales with prefill length, and model attention over very early content degrades at extreme lengths. RAG uses a smaller, targeted context — the chunks that actually matter — rather than the entire corpus. And as the closing section of this chapter explains, this also has important implications for what can and cannot benefit from the prompt cache.

Notice the prompt-assembly node. Its segments are stacked exactly as in Ch 7's anatomy diagram: system prompt and stable documents are amber — cached across requests — while retrieved chunks and the user question are grey — per-query and uncached. We will return to this in the closing section.

The index phase is slow and offline; the query phase is fast and online. This asymmetry is intentional. You can afford expensive preprocessing at index time — running a model to generate contextual blurbs, building multiple indexes for hybrid retrieval — because those costs are amortized across every query ever served. At query time, every millisecond counts.

## Chunking — the unsexy thing that matters most

Retrieval can only surface what indexing captures. Chunking determines what gets indexed. It is the most impactful and most underestimated decision in a RAG system.

The core tension: **chunks too small** lose context (a sentence like "it was discontinued in 2019" is meaningless without knowing what "it" refers to); **chunks too large** bury the relevant signal in noise and push the model to ignore most of the text.

**Fixed-size chunking** splits every N tokens, with optional overlap between adjacent chunks. Simple to implement, works acceptably for homogenous prose — but frequently splits sentences and paragraphs mid-thought. Fine as a baseline; almost always beatable.

**Semantic / recursive chunking** splits on document-aware boundaries: try paragraphs first, then sentences, then words, only reducing granularity when a unit exceeds the target size. Preserves sentence and paragraph coherence. The default in most open-source RAG libraries.

**Hierarchical (parent-child) chunking** indexes small chunks (individual paragraphs) for retrieval precision, then retrieves their parent (the full section) as model context. The granular chunk is easy to match; the parent provides coherence. Higher recall without context loss, but more complex to implement.

**Contextual Retrieval — the Anthropic-blessed pattern.** Before embedding each chunk, a model prepends a short (50–100 token) context blurb explaining what the chunk is about in the context of its source document. A chunk like "Revenue grew by 3% year-over-year" becomes "From Acme Corp's Q3 2024 earnings report (the quarter ending September 2024): Revenue grew by 3% year-over-year." The contextualized version produces a much more specific vector. [Anthropic's research](https://www.anthropic.com/research/contextual-retrieval) shows this reduces retrieval failure rates by 35% with contextual embeddings alone, 49% combined with BM25, and 67% when reranking is added — compared to a plain-embeddings baseline with a 5.7% failure rate.

## Retrieval beyond pure vector search

The vector similarity search from the previous chapter is a strong baseline. In practice, layering additional signals consistently improves quality.

**Hybrid retrieval (dense + BM25):** runs both a vector index and a keyword (BM25) index, merges the ranked results. Dense retrieval generalizes across meaning; BM25 is literal — it catches exact phrase matches that embeddings can miss (proper nouns, error codes, product identifiers). Hybrid is the practical default for production.

**Reranking with a cross-encoder:** takes the top-K ANN candidates and rescores them with a model that sees the query and the chunk together. ANN retrieves the top 150 candidates; the reranker returns the top 20. Services like Cohere Rerank and Voyage Rerank add ~100–200ms latency for meaningful quality gains.

**Query rewriting and HyDE:** transform the user's question before embedding. Query rewriting paraphrases casual queries into the formal register of documents. HyDE generates a hypothetical document that would answer the question, then embeds that document-like text. Both add one extra model call of latency; both help when user vocabulary diverges sharply from document vocabulary.

## Failure modes you will encounter in production

<div class="failure-list">
  <div class="failure-item">
    <strong>Retrieval miss.</strong> The right information exists in the corpus, but the top-K retrieval doesn't include it. Common causes: chunk too small (relevant text spans multiple chunks; no single chunk scores well against the query), embedding vocabulary mismatch (query uses "SLA"; the document says "uptime guarantee"), or a missing metadata filter. The model improvises — and the answer is wrong.
  </div>
  <div class="failure-item">
    <strong>Context dilution.</strong> With 20 retrieved chunks in the prompt, the model has a lot of material. Research consistently shows that models attend less reliably to content in the middle of long contexts. A relevant chunk at rank 12 of 20 may be effectively ignored. Mitigations: rerank to ensure best chunks rank first; limit top-K to 5–10 for most tasks.
  </div>
  <div class="failure-item">
    <strong>Citation drift.</strong> The model cites a chunk that doesn't actually support the claim it's attached to. Anthropic's <a href="https://platform.claude.com/docs/en/docs/build-with-claude/citations">citations feature</a> verifies that cited text exists in the source document — but the model can still associate a claim with a real passage that doesn't quite say what the model claims it says. Programmatic spot-checking is still needed.
  </div>
  <div class="failure-item">
    <strong>Stale index.</strong> Your knowledge base changed; the index didn't. A document was updated or superseded, but the old chunk still lives in the vector store. Treat the index as a data pipeline, not a one-time setup. Add a <code>last_indexed</code> timestamp to every chunk's metadata.
  </div>
</div>

## Common misconceptions

<div class="misconception">
  <div class="misconception-label">Misconception: "RAG fixes hallucinations"</div>
  No. RAG <em>reduces</em> the conditions under which hallucination occurs by giving the model something true to anchor on. The model can still hallucinate a claim within a retrieved chunk, hallucinate a synthesis across chunks, or ignore chunks entirely and pattern-match from training. Think: reduces hallucination, does not prevent it.
</div>

<div class="misconception">
  <div class="misconception-label">Misconception: "Bigger top-K is always better"</div>
  Past a threshold (usually 10–20 chunks for most models and tasks), additional chunks contribute noise rather than signal. Retrieval recall goes up with K; downstream answer quality does not always follow. Reranking and a reasonable K cap (5–10 for most tasks) produce better results than simply retrieving more.
</div>

<div class="misconception">
  <div class="misconception-label">Misconception: "Long context windows make RAG obsolete"</div>
  Counter-argument is multi-pronged. <strong>Cost:</strong> a 200K-token context filled with your full knowledge base costs the same to process on every request; RAG pays for 5–20 relevant chunks per query. <strong>Latency:</strong> time-to-first-token scales with prefill length. <strong>Attention dilution:</strong> at very long contexts, model attention over early content degrades. <strong>Freshness:</strong> a long-context approach still requires manually deciding what to include; RAG retrieves dynamically. And most critically for this site: <strong>cache reuse.</strong> A stable, well-structured cached prefix preserves cache hits across turns. A 200K-token prompt filled with per-request content cannot be cached. Long contexts and RAG are complementary, not substitutes — and the prompt cache is the clearest reason why.
</div>

## The prompt-cache interaction: the chapter's payoff

Chapter 7 established one rule: **stable content early, mutable content late.** Cache breakpoints go after the stable segments. A cache hit is an exact token-level prefix match up to a breakpoint.

Retrieved chunks are per-query. They differ on every request. They cannot be cached across queries. This creates a real architectural tension when designing a RAG system:

**Option A — large stable knowledge bundle (cacheable).** Pre-select a fixed set of high-value documents for a given assistant and place them in the cached prefix with `cache_control: { type: "ephemeral" }`. Every request pays only for the new question and small incremental retrieval. Cost drops after the first turn. Trade-off: the fixed bundle is a guess about what's always relevant. It can be wrong, and it grows stale.

**Option B — pure per-query retrieval (not cacheable).** Run retrieval fresh for every query. Maximum precision. Zero cache benefit across queries; every request pays full input-token cost for retrieved chunks.

**Option C (the practical optimum) — cache a stable core, append fresh retrieval.** Cache a stable core (e.g., product documentation that rarely changes) with a cache breakpoint after it. Append per-query retrieved chunks after the breakpoint, before the question. The cached prefix is saved; the retrieved chunks are fresh per query. This matches Ch 7's anatomy exactly: system prompt + stable knowledge → cached (amber); per-query chunks + question → uncached (grey).

Concretely (numbers illustrative):

| Segment | Tokens | Cached? |
|---|---|---|
| System prompt | 800 | Yes |
| Stable knowledge bundle (10 docs) | 8,000 | Yes |
| Per-query retrieved chunks (5 × 300 tokens) | 1,500 | No |
| Conversation history | 600 | No |
| New user question | 120 | No |
| **Total** | **11,020** | **8,800 cached** |

On a cache hit: pay 8,800 tokens at cache-read rates plus 2,220 tokens at uncached input rates. The stable bundle makes the cache valuable only if it contains content likely to be relevant to every query. If the stable bundle is generic and the real answers live in per-query retrieved chunks anyway, the caching benefit shrinks — and a narrower stable bundle (or none at all) is the better design.

This is the non-obvious tradeoff that flows directly from Ch 7's "stable content early" rule: a broader stable bundle maximizes cache value but reduces per-query retrieval precision. Most production RAG systems land somewhere in between — a small stable bundle of "always relevant" content plus targeted per-query retrieval for specifics.
