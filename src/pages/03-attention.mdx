## How tokens look at each other

After tokenization and embedding, the model has a sequence of vectors — one per token. But so far each token only knows about itself. Attention is the mechanism that lets every token gather information from other tokens in the sequence.

The core idea: each token gets to "ask a question" and then look up answers in the other tokens. The answer isn't a single token — it's a weighted blend of information from whichever earlier tokens seem most relevant to the question. This weighted mixture becomes the output that flows to the next layer.

## Q, K, V — query, key, value

Attention uses three ingredients computed from each token's current representation:

- **Query (Q):** the question a token is asking. *"What context do I need right now?"*
- **Key (K):** what a token is advertising about itself. *"Here is what I represent."*
- **Value (V):** the actual information a token is ready to share. *"Here is what to take from me if you attend to me."*

To compute how much token *i* attends to token *j*: take the query of *i* and compare it (dot product) with the key of *j*. A high dot product means *j* is a good match for *i*'s question. After normalizing those scores with a softmax, you get attention weights that sum to 1. The output for token *i* is then the weighted sum of all the values — taking more from tokens with high weights.

Think of it like a soft search: the query is your search term, the keys are document titles, and the values are the document contents. You retrieve a blend of documents weighted by how well each title matches your search.

**Important:** Q, K, and V are *not* fixed properties stamped onto a token once and for all. They are computed by linear projections (learned weight matrices) applied to each token's representation *at a specific layer*. At layer 1, the representation feeding those projections comes from the embedding. At layer 2, it comes from the output of layer 1. Each layer produces its own Q, K, and V from its own inputs. A token's K and V at layer 3 are different numbers from its K and V at layer 2.

## The causal mask

There is one hard constraint: a token can only attend to tokens that came before it (or itself). It cannot look at tokens that haven't been generated yet. This is the **causal mask** — the upper-right triangle of the attention matrix is zeroed out before softmax. The result: at position *i*, only positions 0 through *i* contribute to the output.

This constraint is what makes the transformer useful for generation: the model is trained on sequences where each position can only see its own past, so at inference time, generating one new token at a time is consistent with what the model learned.

## What attention is *not*

A common misconception: K and V are "properties of a token" — a fixed fingerprint you look up by token ID. **This is wrong.** K and V are computed fresh at every layer from that layer's input representation, which is itself shaped by everything that came before in the layer stack. The same surface token (say, "the") will have different K and V values at layer 1 vs. layer 5, and different values again when it appears in a different sentence. They are layer-specific and context-specific computations, not stored token attributes.

<CacheCallout>
**K and V are layer-specific — this is the key insight for caching.**

At each layer, Q, K, and V are computed by projecting that layer's input representation through learned weight matrices. The same token produces different K and V values at different layers — because each layer's input is a different transformation of the original embedding. K and V are *not* fixed properties of a token.

Here is what *is* stable: within a single generation pass, once the model has processed the prompt tokens through layer L, the K and V values for those tokens at layer L are fixed. They won't change as the model generates new tokens one by one. That is the property the **KV cache** exploits (covered in Ch 6): instead of recomputing every prompt token's K and V at every layer on every generation step, the model stores them after the first pass and reuses them.

The implication for Claude Code's prompt cache (Ch 7): when the prefix of your request — the system prompt, tool definitions, files, and conversation history — stays identical across turns, the stored K/V values for those tokens at every layer stay valid and can be reused across separate API requests, not just within one generation pass. Change even one token in that prefix and the cache goes cold from that point forward.
</CacheCallout>
